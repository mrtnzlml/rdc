# Rossum Local — macOS desktop app for non-developers (design)

Status: draft
Date: 2026-05-26

## Overview

A minimalistic, signed macOS desktop application that pulls one or more Rossum organizations into local folders so they can be read by Claude (Claude Code in a terminal, the claude.ai web app via Project attachments, or the Claude desktop app). The app wraps the existing `rdc` Rust library; it does not reimplement Rossum's pull, snapshot, slug, or `_index.md` logic. It is pull-only — no push, no deploy. v1 ships macOS-only (Apple Silicon + Intel universal binary).

Target persona: non-developers. Two cohorts use the same app, same UI:

- **Internal Rossum SAs / support staff** who already have org access and want to point Claude at customer configs without learning the CLI.
- **External customer admins** who run a Rossum tenant via the web UI, do not use a terminal, and want Claude to help them understand their own configuration.

The app exists because the existing `rdc` CLI assumes engineering fluency — terminals, git, Homebrew, file paths, JSON editing. The desktop app removes those preconditions. The on-disk output is identical to what `rdc init && rdc sync main` produces today, so anything Claude (or a developer) can do with an rdc snapshot works without modification.

## Success criteria

1. A first-time user installs the signed DMG, double-clicks to launch (no Gatekeeper warning), clicks **Add Connection**, enters Rossum URL + Org ID + (email+password or API token), clicks **Add & Sync**, and ends up with `~/Documents/Rossum/<name>/` populated with the full rdc snapshot. Total time from download to working snapshot: under 2 minutes.
2. The synced folder contains the same files in the same layout as `rdc init && rdc sync main` produces today, including `CLAUDE.md`, `README.md`, `envs/main/_index.md`, and per-object JSON + `.py` files.
3. Clicking **Reveal in Finder** opens the folder; clicking **Copy path** copies a plain `~/Documents/Rossum/<name>` string. Both `claude` (Code) opening that folder and dragging the folder into a claude.ai Project work without extra steps.
4. Manual **Sync now** pulls fresh data on demand and updates the visible last-sync timestamp. No background sync, no scheduled sync, no continuous watcher in v1.
5. Credentials never live in plaintext on disk. Tokens and (for email+password Connections) passwords are stored in macOS Keychain.
6. A 300-object Rossum org pulls in under 30 seconds on a typical home connection (matches `rdc` spec §14.4).
7. Removing a Connection deletes its Keychain entries and moves the local folder to `~/.Trash` — never `rm -rf`.

## Architecture

A single signed macOS `.app` containing a Rust Tauri 2 binary. The Rust backend depends on `rdc` as a path or git dependency (Cargo workspace) and orchestrates Rossum pulls via rdc's existing building blocks (`api`, `snapshot`, `state`, `paths`, `overlay`, `slug`). The frontend is a small vanilla-TS + CSS view rendered in the system WebView — sidebar + detail, no SPA framework. Offline-first apart from the Rossum API calls themselves; no telemetry; no third-party services.

```
+---------------------------------+
|  Tauri WebView (TS/CSS)         |  UI only
|   +--------+------------------+ |
|   |Sidebar |  Detail pane     | |
|   +--------+------------------+ |
+---------------+-----------------+
                | tauri::invoke()
+---------------v-----------------+
|  Rust backend (Tauri commands)  |
|    app::connections             |  persisted list of Connections
|    app::sync                    |  orchestrates one pull
|    app::keychain                |  Security framework
|    app::settings                |  default folder, app prefs
+---------------+-----------------+
                | direct Rust calls
+---------------v-----------------+
|  rdc (lib)                      |
|    api · snapshot · state ·     |
|    paths · overlay · slug       |
+---------------------------------+
```

### Rust crate layout

A new crate `rossum-local` at the root of this repo (sibling of the existing rdc crate), wired as a Cargo workspace:

```
Cargo.toml                          [workspace]
rdc/                                existing crate, unchanged in structure
  Cargo.toml
  src/...
rossum-local/
  Cargo.toml                        depends on rdc, tauri, security-framework, keyring, ...
  tauri.conf.json
  src/
    main.rs
    lib.rs
    connections/                    registry CRUD
    sync/                           pull orchestrator
    keychain/                       Security framework wrapper
    settings/                       prefs persistence
  ui/
    index.html
    main.ts
    styles.css
  icons/                            generated by `tauri icon`
```

The existing `rdc` crate moves under `rdc/` to become a workspace member; its public API and on-disk path are unchanged. The desktop app depends on it via `rdc = { path = "../rdc" }`.

### App name and identifiers

- App name: **Rossum Local** (placeholder; trivial to rebrand).
- macOS bundle identifier: `ai.rossum.local`.
- Keychain service: `ai.rossum.local`.
- Application Support directory: `~/Library/Application Support/ai.rossum.local/`.

The "Local" suffix conveys the value proposition (pulls Rossum *locally*) and avoids naming collision with any cloud service or claude.ai workspace.

## On-disk layout

### Per-Connection folder

```
~/Documents/Rossum/<connection-slug>/
├── CLAUDE.md                       rdc-generated agent guide (entry point for Claude)
├── README.md                       rdc-generated human-facing README
├── rdc.toml                        [envs.main] api_base, org_id
├── .gitignore                      ignores secrets/ and .rdc/
└── envs/main/
    ├── _index.md                   rdc-generated map (regenerated every sync)
    ├── organization.json
    ├── workspaces/<ws>/queues/<q>/{queue,schema,inbox}.json
    ├── workspaces/<ws>/queues/<q>/formulas/<field>.py
    ├── workspaces/<ws>/queues/<q>/email-templates/<slug>.json
    ├── hooks/<slug>.{json,py,js}
    ├── rules/<slug>.{json,py}
    ├── labels/<slug>.json
    ├── engines/<engine-slug>/{engine.json,fields/<field>.json}
    ├── workflows/<wf-slug>/{workflow.json,steps/<step>.json}
    └── mdh/<dataset>/{collection.json,indexes.json}
```

The env name is always `main`. Reasoning: every Connection is a single-org snapshot; uniformity makes paths predictable for non-developers who never type them. If a user wants test vs prod side-by-side, they create two Connections (e.g., `acme-test` and `acme-prod`), each with its own `envs/main/`.

### Folder naming

The folder name is a slug derived from the Connection's user-visible name via `rdc::slug::slugify` (already exists). The function lowercases, replaces non-alphanumeric ASCII runs with `-`, and drops non-ASCII characters — so `Faktura č. 1` becomes `faktura-1` and `Acme Corp — Production` becomes `acme-corp-production`. Folder paths are always ASCII-safe, even when Connection names contain Cyrillic, Czech, or Chinese. The Unicode name is preserved verbatim in the registry's `name` field and shown in the sidebar; only the on-disk folder slug is ASCII. On slug collision between two Connections, the new one's slug is suffixed via `rdc::slug::slugify_unique` (`-2`, `-3`, etc.); the existing Connection's slug is unchanged.

### Connection registry

A small JSON file at `~/Library/Application Support/ai.rossum.local/connections.json`:

```json
{
  "version": 1,
  "connections": [
    {
      "id": "01J5K7...",
      "name": "Acme Corp — Production",
      "slug": "acme-corp-production",
      "api_base": "https://acme.app.rossum.ai/api/v1",
      "org_id": 12345,
      "folder": "/Users/alice/Documents/Rossum/acme-corp-production",
      "auth_kind": "password",
      "last_sync_unix": 1763500000,
      "last_status": "ok",
      "file_count": 287
    }
  ]
}
```

- `id` is a ULID generated at Add time; it is the Keychain `account` for that Connection's credentials and never changes.
- `auth_kind` is `"token"` or `"password"`.
- `last_status` is `"ok"`, `"error:<short message>"`, or `"never"`.
- No secrets in this file. It is safe to back up, sync via iCloud Drive, etc.

The file is written atomically (temp + rename, same pattern rdc uses). Reads tolerate a missing file (treat as empty registry).

### App settings

A second small file `~/Library/Application Support/ai.rossum.local/settings.json`:

```json
{
  "version": 1,
  "default_folder_parent": "/Users/alice/Documents/Rossum",
  "update_channel": "stable"
}
```

`default_folder_parent` is the parent directory under which new Connections are placed. Users can change it in `App ▸ Settings…`. Existing Connection folders are not moved.

## Secrets and Keychain

All credentials live in macOS Keychain. The synced folder never contains a token.

| Keychain entry | service | account | data |
|---|---|---|---|
| Token (any auth mode) | `ai.rossum.local` | `<connection-id>` | `{token, expires_at_unix}` JSON |
| Username (password auth) | `ai.rossum.local.username` | `<connection-id>` | plain string |
| Password (password auth) | `ai.rossum.local.password` | `<connection-id>` | plain string |

Each entry uses a generic password Keychain item via the `security-framework` crate. ACLs default to "this app only" — only the signed `Rossum Local` binary can read them. After a code-signing identity change (e.g., switching Apple Team IDs), users re-authenticate. This is acceptable because re-auth is cheap (one sheet) and only happens on Team-ID changes, which are operational decisions.

The desktop app does not share Keychain with the `rdc` CLI. The CLI continues to use its existing `secrets/<env>.secrets.json` + env-var resolution. The `secrets/` directory inside a Connection folder stays empty and gitignored. If a user later runs `rdc` directly in a Connection folder, they set `RDC_TOKEN_MAIN` or run `rdc auth main` — the desktop app's Keychain is opaque to the CLI. This is a deliberate v1 choice; the analysis behind it is in the "Why CLI doesn't share Keychain" subsection below.

### Why CLI doesn't share Keychain in v1

macOS Keychain ACLs are scoped by code signature. The desktop app is signed with Apple Developer ID. The Homebrew-distributed `rdc` CLI is not signed under that identity. Sharing a Keychain entry between them would require either:

1. Signing the `rdc` CLI release binaries under the same Team ID, enrolling them in the Apple Developer Program, notarizing each release. Real ops change to rdc's release pipeline.
2. Granting per-binary ACL exceptions, which prompts the user on first access from each binary hash. The hash changes after every `brew upgrade rdc`, so the prompt recurs.

Both options have costs disproportionate to the value: rdc CLI users today are happy with the secrets file; the desktop app's users will not also use the CLI. Revisit if/when rdc gets a signed release pipeline.

```
|               [ Change… ]                    |
|                                              |
|            [ Cancel ]      [ Add & Sync ]    |

```

- The two auth radio buttons swap the input cluster: token mode shows a single Token field; email+password mode shows Username and Password fields.
- Token field uses `type="password"` (masked) with a reveal toggle.
- The Folder field shows the slug-derived default; "Change…" opens a folder picker. The picker validates the chosen path is writable and not inside another Connection's folder.
- "Add & Sync" is disabled until Name, API URL, Org ID, and credential fields are filled. On click:
  1. Normalize the API URL (strip trailing slash, ensure `/api/v1` suffix if missing — common user error).
  2. Validate credentials by calling `GET /v1/organizations/<org_id>` with the token (or after `POST /v1/auth/login` for email+password).

### Edit credentials sheet

A smaller variant of Add Connection containing only the auth mode radio and the credential fields. The current `auth_kind` is pre-selected; the Connection's name and URL are read-only at the top of the sheet for context.

Credential field pre-population follows OS-keychain norms — secrets are never re-displayed:

- **Token mode:** Token field is empty with placeholder text "Enter new token". The user must paste a new token to save; saving with an empty field is disabled.
- **Email + password mode:** Username field is pre-filled with the stored username (it is not a secret). Password field is empty with placeholder "Enter new password". The user must enter a password to save.

The user can switch `auth_kind` (token ↔ password) in the same sheet; doing so swaps the input cluster. On save, the app re-validates the new credentials against `GET /v1/organizations/<org_id>` (after `POST /v1/auth/login` for password mode) before overwriting Keychain. A typo never replaces a working credential silently — the old Keychain entries remain until the new ones validate.

## Sync orchestration

### One sync = one rdc pull-only pass

Backend command `app::sync::run(connection_id)`:

1. Look up Connection in registry; resolve token from Keychain.
   - If token's `expires_at_unix` is past, or within 60 seconds of expiry, and `auth_kind == "password"`: call `POST /v1/auth/login` silently with the stored username + password, update the Keychain token entry, continue. If `auth_kind == "token"` and the token is expired, fail with an `Auth(Expired)` error that the UI surfaces as the "Sign-in expired" banner.
2. Reconcile `<folder>/rdc.toml` with the Connection's `api_base` and `org_id`. Write if missing; update if the user edited the Connection details via Edit Connection.
3. Build rdc's `RossumClient` via `rdc::api::RossumClient::new(api_base, token)?` with the in-memory token (never write `secrets/main.secrets.json`).
4. Invoke rdc's pull orchestrator in no-push, no-prompts mode:
   - Fetch every in-scope object concurrently (rdc's `api::rate_limit` governs).
   - Decode into `rdc::model` structs.
   - Write per-object JSON + extracted `.py` files atomically via `rdc::snapshot::writer::write_atomic`.
   - Update `.rdc/state/main.lock.json` via `rdc::state::Lockfile`.
   - Regenerate `envs/main/_index.md` via `rdc::cli::index::generate`.
   - Never call any write endpoint; never invoke any `inquire` prompt.
5. Stream progress events back to the WebView via Tauri's event system on a `sync-progress` channel. Event payload: `{connection_id, phase: "auth" | "fetch" | "write" | "index" | "done", count_done, count_total, current_label}`.
6. On completion: update `last_sync_unix`, `last_status`, `file_count` in the registry. Save atomically.
7. On error: update `last_status` to `"error:<short message>"`, leave `last_sync_unix` unchanged, emit a final `sync-progress` event with `phase: "error"` and the message.

### rdc integration shape

`rdc::cli::sync::run` is unsuitable for direct embedding — it prints via `Log` and uses `inquire` for prompts. The desktop app calls one level lower, invoking the building blocks directly:

- `rdc::cli::pull::*` for fetch.
- `rdc::cli::index::generate(paths, lockfile)` for `_index.md`.
- `rdc::state::Lockfile::load` / `::save` for the lockfile (atomic write is internal to `save`).
- `rdc::snapshot::writer::write_atomic(path, bytes)` for individual files (free function, not a method).

The orchestrator the desktop app writes is approximately 150–250 lines because the building blocks already do the work.

**rdc-side change needed:** some functions inside `rdc::cli::pull` and `rdc::cli::index` are currently `pub(crate)` or implicit-module-private and will need to be promoted to `pub`. The implementation plan will enumerate the exact symbols. No structural refactor; only visibility changes. This is the only rdc-side modification anticipated.

### Concurrency

- One sync at a time per Connection (enforced via a `tokio::sync::Mutex` keyed by Connection ID inside `app::sync`).
- Across Connections, multiple syncs may run in parallel as independent `tokio::task::spawn` tasks. The UI shows progress only for the currently-selected Connection; other syncs run silently and update their status badges in the sidebar when done.
- rdc's existing API rate limiting (`api::rate_limit`) governs per-client throughput.

### Cancellation

Each sync task carries a `tokio_util::sync::CancellationToken`. The detail pane shows a "Stop sync" button while a sync is in progress; clicking it cancels the token. The task drops in-flight HTTP futures cleanly. rdc's atomic writes guarantee no half-written files; the partial snapshot is left in a consistent state corresponding to whatever fetches completed before cancellation. The next sync picks up cleanly.

## Authentication flow

### Token mode (Add or Edit)

1. User enters token in the sheet.
2. App constructs `rdc::api::Client` and calls `GET /v1/organizations/<org_id>`.
3. On 200: write Keychain entry `{token, expires_at_unix: null}` for service `ai.rossum.local`, account `<connection-id>`. The token has no app-known expiry; rdc handles a mid-call 401 by surfacing it for UI handling.
4. On 401/403: inline error in the Token field; no Keychain write.

### Email + password mode (Add or Edit)

1. User enters username and password.
2. App calls `POST /v1/auth/login` with `{username, password}`.
3. On 200: response `{key, domain}`. App validates by calling `GET /v1/organizations/<org_id>` with the new token (catches the rare case of a token that authenticates but lacks access to the org).
4. On both calls succeeding: write three Keychain entries — token (with `expires_at_unix = now + 162 * 3600`, matching rdc's documented default), username, password.
5. On any failure: inline error; no Keychain writes.

### Silent re-login

Before every API call inside `app::sync::run`, the orchestrator checks the cached token's `expires_at_unix`. If expired or within 60 seconds of expiry:

- If `auth_kind == "password"`: call `POST /v1/auth/login` with the stored credentials, update the Keychain token entry, continue.
- If `auth_kind == "token"`: do nothing (the token has no app-known expiry; rely on the mid-call 401 handler).

If a sync receives a 401 mid-call:

- If `auth_kind == "password"`: silent re-login, retry the failing call once.
- If `auth_kind == "token"`: stop the sync; emit `phase: "error"` with `"sign-in expired"`. The detail pane shows the "Sign-in expired" banner and an "Edit credentials" button that opens the Edit credentials sheet with the Token field empty and focused (per the sheet's standard behavior; see UI § Edit credentials sheet).

This mirrors rdc's own silent re-login logic from `src/cli/auth.rs::refresh_token_for_401`.

## Error handling

| Condition | UX |
|---|---|
| Invalid token on Add | Field shakes red; inline "Sign-in failed. Check your token and try again." |
| Wrong org id | Inline "Organization 12345 not found on this URL. Check the API URL and Org ID." |
| Token expired mid-sync (token-only auth) | Sync stops; detail-pane banner "Sign-in expired" + "Edit credentials" button |
| Token expired + password creds in Keychain | Silent re-login (no UI event); sync continues |
| Network offline | Sync button click shows toast "No internet connection. Try again when you're back online." |
| 429 from Rossum | rdc's `api::retry` handles transparently; UI status line shows "Waiting…" longer |
| 5xx persistent (after rdc's retries) | Sync error: "Rossum is having trouble. Try again in a few minutes." |
| Disk full / write fails | Sync error names the path: "Couldn't write to ~/Documents/Rossum/…. Make space and retry." |
| Folder deleted by user mid-app | On next sync: detect missing folder; sheet "Folder is gone. Re-create it or Remove this Connection?" |
| Keychain access denied (Team ID change) | Sheet "Couldn't read your saved sign-in. Edit credentials to re-enter." |
| User edited `rdc.toml` by hand | App re-reads on next sync; on invalid TOML, sheet "Configuration file is broken. Re-create from your saved settings?" with one-click rebuild |
| User opens same Connection on a second Mac | Each Mac has its own Keychain. The user re-auths once on the second machine. Folder contents can be copied or re-synced. |
| Rossum org has no MDH | rdc handles natively; `envs/main/mdh/` simply absent. No UI difference. |
| API URL has trailing `/` or missing `/api/v1` | Add Connection normalizes before save |
| Concurrent rdc CLI use on same folder | rdc's `fs4`-based lockfile prevents corruption; app surfaces "Folder is in use by another rdc process. Try again." |

### Crash recovery

Rust panics in the backend are caught by Tauri; the app shows a "Something went wrong" sheet with "Copy diagnostics" (timestamps + last sync log, no secrets) and Quit / Restart buttons. The synced folder is never in a half-written state because rdc's writes are atomic (temp + rename).

## Distribution and updates

### Signing and notarization

- Apple Developer ID Application certificate.
- Build: `tauri build --target universal-apple-darwin` produces one universal binary covering Apple Silicon and Intel.
- `codesign --options runtime` with hardened runtime enabled.
- `notarytool submit` against Apple's notarization service.
- `stapler staple` the resulting `.dmg`.
- Output: a signed, notarized, stapled `.dmg` of approximately 25 MB.
- First launch on any Mac: Gatekeeper passes silently; no right-click-Open workaround.

### Not in Mac App Store

The sandbox would require a folder-picker prompt every launch for `~/Documents/Rossum/` access (or per-Connection bookmark URLs persisted via NSURL security scoping). Direct distribution sidesteps this. Trade-off accepted: users do not get App Store auto-update, so the app needs its own update path.

### Updates via Tauri updater

Tauri's updater plugin checks a GitHub Releases endpoint on launch (with a 24 h debounce). When an update is available, it shows a non-modal banner with "Install & Restart" and "Later" buttons. Updates are signed with a separate Tauri update key (Ed25519); the public key ships in the app binary. No silent updates; user always confirms.

For internal-only distribution (e.g., Rossum staging the app through an internal portal first), the updater feed URL is settable via env var at build time, pointing at an internal CDN.

### Release pipeline

A new GitHub Actions workflow in this repo, triggered on tags matching `desktop-v*` (e.g., `desktop-v0.1.0`):

- Runs on `macos-14` (Apple Silicon).
- Builds the universal binary via `tauri build --target universal-apple-darwin`.
- Signs with secrets from the repo's GitHub Environment: Apple Developer Team ID, signing .p12, notarization App Store Connect API key, Tauri update key.
- Uploads `.dmg`, `.dmg.sig`, and `latest.json` to the GitHub Release.

The existing rdc CLI release pipeline (Homebrew tap, GitHub Releases for `rdc-*.tar.gz`) is untouched. The two release pipelines are independent.

### Versioning

The desktop app's version is independent from rdc's version. The Cargo workspace pins the rdc dependency exactly (e.g., `rdc = { path = "../rdc", version = "=0.1.1" }`), bumped together with the app when needed. `App ▸ About` displays both versions so support can correlate bug reports.

### Homepage

A README section at the root of this repo (under a new "Desktop app" heading) with a "Download Rossum Local (macOS)" link to the latest GitHub Release. No separate landing page in v1.

### Apple Developer Program

$99/year, required for Developer ID + notarization. Rossum-side decision: which Team ID to use and who pays. This design assumes a Team ID is available before v0.1.0 ships. If not, v0.1.0 ships unsigned for internal-only testing and notarization slips to v0.2.0.

## Testing

### Rust unit tests (`cargo test`)

- `app::connections`: registry serde round-trip; add / remove / edit; name collision handling (slug suffixing).
- `app::keychain`: token write / read / delete via `security-framework`. A `#[cfg(test)]` feature swaps the live Keychain for an in-memory `HashMap` for headless CI runs. Live Keychain tests are gated to `cargo test -- --ignored` and run only on the developer's machine.
- `app::sync`: orchestrator driven by `wiremock` (already a rdc dev-dep) simulating Rossum. Cases: happy-path one Connection, mid-sync cancel, network error, 401 with password creds (silent re-login), 401 with token only (banner emitted), 429 rate-limit retry, 5xx persistent.
- URL normalization for Add Connection (trailing slash, missing `/api/v1`, plain hostname).
- `rdc.toml` reconciliation when user edits `api_base` in Edit Connection.

### Tauri integration tests

Tauri ships `tauri::test`. One end-to-end test per major flow, driven from Rust:

- Add Connection → first sync → file count matches mock fixture.
- Sync existing Connection → progress events stream → completion event.
- Edit credentials → re-validate → Keychain entries replaced atomically.
- Remove Connection → Keychain cleared, folder moved to `~/.Trash` (verified via `NSFileManager`-equivalent check).

### Frontend tests

A small Playwright suite running against `tauri dev`. Three flows:

- Empty-state → Add Connection → first sync completes; status line updates.
- Two Connections, switching sidebar selection mid-sync; progress for the un-selected Connection continues silently and reflects in its sidebar badge.
- Injected 401 mid-sync → error banner appears with "Edit credentials" button.

### Manual visual smoke checklist

Run before each release tag:

- Light mode rendering correct.
- Dark mode rendering correct.
- Sidebar drag-resize works.
- Window resize down to minimum size; no clipped content.
- Long Connection names truncate with ellipsis in the sidebar.
- Non-ASCII Connection names (Cyrillic, Czech, Chinese) round-trip through folder paths and registry serialization.

### CI matrix

GitHub Actions on `macos-13` (Intel) and `macos-14` (Apple Silicon) for:

- `cargo test` on every PR.
- Universal binary smoke build on every PR.
- Full sign + notarize + DMG on tag builds only (notarization is slow and rate-limited).

### Performance acceptance bar

- A 300-object Rossum org pulls in under 30 seconds on a typical home connection (matches rdc spec §14.4).
- Tauri startup under 1 second on Apple Silicon, under 2 seconds on Intel.
- Memory footprint under 100 MB resident with five Connections idle.

## Backwards compatibility

The desktop app is additive. The existing `rdc` CLI, its `rdc.toml` schema, its on-disk snapshot layout, and its secrets file format are unchanged. The only rdc-side modifications are visibility promotions (`pub(crate)` → `pub`) on a handful of pull and index helper functions, enumerated in the implementation plan. No behavior changes; no breaking changes for existing rdc users.

A user could use the desktop app to bootstrap a Connection folder, then point `rdc` at it from the terminal — `rdc auth main --token …` would write `secrets/main.secrets.json` and the CLI would work normally from there. The app's Keychain entry is opaque to the CLI; the CLI's secrets file is opaque to the app. They coexist without interference.

## Out of scope (v1)

- Pushing changes back to Rossum.
- Cross-environment deploy.
- Multiple envs inside one Connection folder. Workaround: create one Connection per env.
- Conflict resolution UI (the app never pushes, so no conflicts exist).
- Watch mode or scheduled sync.
- Built-in Claude chat or any LLM call from inside the app.
- Linux and Windows builds. Tauri can target both from the same Rust code; not committed for v1.
- Mac App Store distribution.
- Sharing Connection folders between Macs via the app. Users can manually copy a folder if they want.
- iOS or iPadOS support.
- "Open in Claude Code" terminal launcher (user requested it be omitted).
- Sharing Keychain credentials with the rdc CLI.

## Open questions

1. **Apple Developer Team ID.** Which Rossum Team ID is used for code signing and notarization? This is a Rossum-side operational decision and not blocking the implementation plan, but is blocking the first signed release.
2. **App name.** "Rossum Local" is a placeholder. Final name (and any trademark or brand-approval steps) is a Rossum-side decision. Renaming is a global find-and-replace in `tauri.conf.json`, the bundle identifier, the Keychain service, and the Application Support directory — straightforward and reversible.
3. **Icon and visual identity.** v1 ships with a placeholder icon (the rdc CLI's amber accent palette, app-shaped). Final icon design is deferred.
